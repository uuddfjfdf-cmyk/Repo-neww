OAuth 2.0 Flow
Cara kerja authorization code flow

🔄 Authorization Code Flow

Flow paling aman. App kamu minta izin → dapat code → tukar ke token. Code hanya sekali pakai dan expire cepat.

🔑 Scope yang diminta

Izin spesifik: email profile openid. Jangan minta izin lebih dari yang dibutuhkan!

🛡️ State Parameter

Random string untuk mencegah CSRF attack. Wajib divalidasi saat callback datang.

// URL redirect ke Google OAuth
const url = `https://accounts.google.com/o/oauth2/v2/auth?

⏱️ Time-based OTP (TOTP)

Kode berubah setiap 30/60 detik. Dihitung dari: HMAC(secret, floor(time/30)). Dipakai di 2FA apps (Google Authenticator).

📨 SMS/Email OTP

Server generate kode random 6 digit, simpan di DB dengan expiry, kirim via SMS/email. Lebih sederhana tapi butuh provider eksternal.

// Generate OTP 6 digit
const otp = Math.floor(100000 + Math.random() * 900000);
const expires = Date.now() + 10 * 60 * 1000; // 10 mnt

// Verifikasi OTP
if (stored.otp !== input || Date.now() > stored.expires)
return 'OTP tidak valid atau expired';

⚠️ Penting tentang JWT

Payload JWT bisa dibaca siapapun — jangan simpan password atau data sensitif di sini! JWT hanya "ditandatangani" bukan dienkripsi. Gunakan HTTPS selalu.

🗺️ Mana yang Harus Dipilih?
Perbandingan cepat metode autentikasi

✍️ Email + Password

Paling umum. User perlu ingat password. Wajib pakai bcrypt, HTTPS, dan pertimbangkan 2FA. Cocok untuk hampir semua app.

🌐 OAuth / Social Login

Paling mudah buat user. Tidak perlu manage password sama sekali. Cocok kalau target user pasti punya Google/GitHub.

✨ Magic Link

Passwordless & secure. User perlu akses email. UX smooth, cocok untuk SaaS B2B atau tool internal tim.

📱 OTP SMS/Email

Bagus untuk 2FA atau verifikasi nomor HP. Biaya SMS bisa tinggi. Email OTP gratis tapi masuk spam kadang.
